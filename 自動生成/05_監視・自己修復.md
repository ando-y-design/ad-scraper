# 工程⑤　監視・自己修復（Watchdog）

## 概要

`watchdog_worker` が**30秒ごと**にループし、スレッドの死活監視・自動再起動・設定再読み込み・自己修復を担う。

---

## Watchdog ループのタスク一覧

| 頻度 | タスク |
|------|-------|
| 毎ループ（30秒） | ① スレッド死亡検知 → 再起動 |
| 毎ループ（30秒） | ② ハング検知（120秒超 → ゾンビ強制置換） |
| 毎ループ（30秒） | ③ `config.json` 再読み込み（ランタイム変更反映） |
| 毎ループ（30秒） | ④ `control.json` 処理（pause/stop/キーワード追加） |
| 毎ループ（30秒） | ⑤ キーワード自動補充（枯渇時にアーカイブを再有効化） |
| 10分ごと | ⑥ DBゴミクリーニング（db_janitor） |
| 5分ごと | ⑦ status.html 更新（localhost:8080） |
| 30分ごと | ⑧ 自己修復サイクル（diagnostics → repairer） |
| 2時間ごと | ⑨ AutoTuner（delay/cooling を自動最適化） |
| 1日1回 | ⑩ 競合代理店事例スクレイプ |

## スレッド死活監視とハートビート

```
各ワーカースレッド
    └── beat(name) を定期呼び出し
              │ state._heartbeat[name] = time.time()
              ▼
watchdog（30秒ごと）
    └── now - _heartbeat[name] を計算
          ├─ > 300秒 → ハング警告ログ
          ├─ > 120秒 → Chrome強制終了 + 新スレッド起動
          └─ is_alive() == False → 再起動
```

### 再起動バックオフ

1時間あたりの再起動が `max_restart_per_hour`（デフォルト3回）を超えると指数バックオフ：

| 超過回数 | 待機時間 |
|---------|---------|
| +1回 | 60秒 |
| +2回 | 120秒 |
| +3回 | 240秒 |
| ... | 最大1800秒（30分） |

## control.json によるランタイム制御

watchdog が毎ループで `control.json` を読み込み、以下を処理する：

```json
{
  "status": "running",      // "running" / "paused" / "stopped"
  "pause": false,
  "stop": false,
  "add_keywords": ["新キーワード1", "新キーワード2"],
  "remove_keywords": ["削除キーワード"]
}
```

| status 値 | 動作 |
|---------|------|
| `running` | 通常稼働（pause 解除も自動） |
| `paused` | 全ワーカーを一時停止（watchdog は継続） |
| `stopped` | シャットダウン開始（全スレッドに終了通知） |

`add_keywords` / `remove_keywords` は処理後に自動で空配列にリセットされる。

## 自己修復サイクル（30分ごと）

```
diagnostics.py
    │ メトリクス収集
    │ - 直近N件の成功率
    │ - 会社名空欄率
    │ - 電話番号空欄率
    │ - スレッドハング頻度
    ▼
問題検知?
    ├─ なし → スキップ
    └─ あり → pending_fixes.jsonl に記録
                    │
                    ▼
               repair_worker.py
                    │ repairer.py 経由で Claude CLI 呼び出し
                    │ コード修正プロンプトを生成
                    ▼
               修復成功？
                    ├─ 成功 → 対象スレッドをリロード・再起動
                    └─ 失敗 → repair_history.jsonl に記録
```

> **注意**: `self_repair/` 配下のコードは手動で変更しないこと。修復ループが再発する危険がある。

## AutoTuner（2時間ごと）

`self_repair/auto_tuner.py` が過去の取得実績を分析し、`config.json` の以下を自動調整する：

| パラメータ | 調整ロジック |
|-----------|------------|
| `min_delay_seconds` | 広告取得成功率が高ければ短縮 |
| `max_delay_seconds` | CAPTCHA発生が多ければ延長 |
| `keyword_cooling_hours` | 新規企業が減ってきたら延長 |

| ファイル | 内容 | サイズ制限 |
|---------|------|----------|
| `logs/main.log` | 全スレッドの動作ログ | 10MB × 5世代 |

| `logs/pending_fixes.jsonl` | 自己修復の候補リスト | — |
| `logs/repair_history.jsonl` | 修復履歴 | — |

## 運用チェックリスト

### 毎日確認

- [ ] `logs/main.log` にERRORが増えていないか
- [ ] Google Sheets の行が増えているか（今日の取得件数）
- [ ] `localhost:8080` でステータス確認

### 週次確認

- [ ] `logs/pending_fixes.jsonl` に未修復の問題がないか
- [ ] キーワードのアーカイブ率（消化しすぎていないか）
- [ ] SerpAPI クォータの残量

### トラブル時の初動

1. `logs/main.log` の最新ERRORを確認
2. `control.json` の status が `running` になっているか確認
3. プロセスが生きているか確認（`tasklist | findstr python`）
4. 死んでいれば `python main.py` で再起動

### 緊急停止

```json
// control.json を編集
{ "status": "stopped" }
// 30秒以内に全スレッドが終了する
```
